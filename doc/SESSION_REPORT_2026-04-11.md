### 2. Исправление критического бага RPC протокола ✅

**Проблема:** Клиент отправлял `VersionNode` вместо модели графа → ошибка десериализации на сервере.

**Корневая причина:**
```dart
// versioned_repository_impl.dart:103
await _storage.put(_collection, nodeId, node.toMap());  // ❌ VersionNode
```

**Решение:** Изменены 2 места в коде:

1. **createEntity (строка 103):**
   ```dart
   await _storage.put(_collection, nodeId, model.toMap());  // ✅ Модель
   ```

2. **updateDraft (строки 186-192):**
   ```dart
   await baseStorage.put(_collection, nodeId, {
     'operation': 'updateDraft',
     'nodeId': nodeId,
     'data': model.toMap(),  // ✅ Модель
   });
   ```

**Результат:**
- ✅ Клиент отправляет модель, сервер получает модель
- ✅ Соответствует принципу "тонкого клиента"
- ✅ Сервер управляет версионированием
- ✅ Нет ошибок десериализации

## 💡 Ключевые решения

### 1. Принцип "Тонкого клиента"

**Решение:** Клиент отправляет данные, сервер управляет версионированием.

**Обоснование:**
- Клиент не генерирует nodeId, status, version
- Сервер полностью контролирует метаданные
- Невозможно подделать версионную информацию
- Упрощает клиентский код

### 2. Механизм специальных операций

**Решение:** Использовать поле `'operation'` в data для маршрутизации RPC.

**Как работает:**
```dart
// Клиент
await storage.put(collection, id, {
  'operation': 'updateDraft',  // ← Специальная операция
  'nodeId': nodeId,
  'data': model.toMap(),
});

// RemoteVaultStorage распознаёт и вызывает
await _rpc(collection, 'updateDraft', cleanData);

// Сервер обрабатывает
case 'updateDraft':
  final model = reg.fromMap(args['data']);
  await repo.updateDraft(args['nodeId'], model);
```
